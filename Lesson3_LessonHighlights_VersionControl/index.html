VCS VCS? You Mean VHS Right?

VCS, Version Control System is a method of software that allows for a system to record changes to a file or files over a set of time- that way simultaneous changes to a document or set of documents won’t cause a collision of data when saved to the same data server. A single server containing all the versioned files, and a number of clients pull these files for editing and use, be it for corporate business, personal business, or home user situations. The methodic set up of this allows for quicker and easier access to files, but along with these upsides comes a steep downside; due to its centralized administrative control of one unit, it’s a single point of control is also its own failure domain- when it goes down there is no way of transferring information anymore.

Distributed Version Control Systems- Sharing for One, is Sharing for All

Embracing the Control Systems

Similar to an SDN controller (software-defined networking), the VCS manages the flow of the network to an intelligible level where open flow control to the server is permitted, and the secondary levels of packet distribution (a packet is a container with requested information, protected in an “envelope”). The VCS, is the core of the centralized point of distribution to all who connect to that network- hence it lies as the core of the network and any communications between applications and the end user devices will be handled via the server device protocols for optimal network path progression. Vendors of the VCS and its controllers include the switches, multilayer 3 switches, routers, and management of the control plane off/on the network hardware.

Centralized Network Command and Control Focal Point of Control

Ensuring that the focal point of control for the network has more than one method of distributing information over the local/remote network is imperative. In regards to failure domains which, when trunking (administrative control enabled on a virtual or physical link) occurs on the switches associated with this, full site outrages occur when a broadcast storm happens over the network (flooding of packets out each port of every available device trying to locate a device that is not on the network, versus sending just one packet). With fate sharing, and failure domains, the domain is bound to a system where failure has little to no impact on its services but what relays the services will suffer; i.e. an example of this from a site story of etheralmind is a car tire coming off, bearing fruit to an inevitable car crash and possibly causing more than cars on the freeway to crash as a direct result of that one failure.
